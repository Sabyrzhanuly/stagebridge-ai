# PG Control Center + AI — сценарий демонстрации (хакатон OpenAI)

**Одной фразой:** веб-система управления парком PostgreSQL (серверы, бэкапы, миграции структуры, диагностика, мониторинг) с **ИИ-слоем на OpenAI**, который объясняет риски и предлагает безопасные действия — прямо в интерфейсе.

---

## Доступы

| Что | Где | Логин |
|---|---|---|
| Приложение | http://localhost | `admin` / `1sZz4KQIuKFCIIZM` |
| API health | http://localhost:8000/api/health | — |
| MinIO консоль | http://localhost:9001 | `minioadmin` / `minioadmin123` |

Ключ OpenAI задаётся в **Настройки → ИИ** (хранится зашифрованным в БД, без `.env`).

## Сценарий (5–7 минут)

Идея показа: сначала «обычная» мощь системы, затем — **ИИ в каждом узле**. Акцент — на ИИ (хакатон OpenAI).

### 1. Дашборд (30 сек) — «это не игрушка»
- Открыть http://localhost. Показать: сервер Demo PostgreSQL **Online**, worker Online, мониторинг Live.
- Тезис: «Полноценная панель управления парком PostgreSQL — серверы, бэкапы, миграции, мониторинг».

### 2. Killer-фича: миграция структуры + ИИ-план (2 мин)
- **Сценарии → Миграция структуры** → сценарий *Demo test>prod* → иконка **истории** (часы).
- Показать **Сгенерированный SQL** (реальный дифф `demo_test → demo_prod`: новая таблица, колонка, constraint, функция).
- Нажать **«✦ ИИ-план миграции: риски и порядок»**.
- ИИ выдаёт: **уровень риска**, конкретные **риски** (unique на существующих данных, дубли), **безопасный порядок** применения и **откат**.
- Тезис: «Дифф считает детерминированный код. ИИ — объясняет риски и строит безопасный план. ИИ ничего не выполняет сам — только советует».

### 3. ИИ-ассистент (1 мин) — «встроенный DBA»
- Кнопка **✦ ИИ** справа снизу (на любом экране).
- Спросить: *«Как безопасно перенести структуру из test в prod без потери данных?»*
- Показать живой ответ на русском (шаги + риски + команды).

### 4. Диагностика + ИИ (1 мин)
- **Demo PostgreSQL → Диагностика → Запустить**.
- Реальные проверки (роли, ACL, триггеры). Нажать **«✦ ИИ-анализ диагностики»**.
- ИИ: severity, что не так, рекомендации.

### 5. Реальный бэкап в S3 + ИИ-риск (1.5 мин) — «полный цикл»
- **Бэкапы → выбрать Demo PostgreSQL**. Показать историю: `demo_shop` — **success, 8.9 KB в MinIO**.
- (Опц.) Открыть MinIO консоль localhost:9001 → bucket `backups` → реальный файл дампа.
- Нажать **«✦ ИИ: риск-анализ бэкапов и restore»**.
- ИИ учитывает реальное состояние: у demo_shop есть копия, у demo_prod/test — нет → **риск**, что проверить.
- Тезис: «Реальный pg_dump → S3, система сама ставит нужный pg-клиент под версию сервера. И ИИ оценивает готовность к restore».

### 6. Ключ из UI (30 сек) — «без секретов в конфигах»
- **Настройки → ИИ**: ключ вводится в интерфейсе, шифруется в БД, работает без перезапуска.

## Тезисы для судей (почему это сильно)

1. **ИИ в правильном месте.** Не «чат ради чата»: детерминированный код считает факты (дифф схем, состояние бэкапов, диагностику), а OpenAI **объясняет риски и предлагает безопасный план** на этих фактах.
2. **Advisory-модель безопасности.** ИИ никогда не выполняет SQL — только советует. Реальные операции идут через контролируемые задачи (Celery), с approval и откатом.
3. **Полный реальный цикл, не мокап.** Реальные PostgreSQL-серверы, реальный pg_dump в S3/MinIO, реальные структурные миграции с verify и атомарным свопом БД.
4. **4 точки ИИ:** план миграции, ассистент, диагностика, риск-анализ бэкапов — единый слой (`ai_service` + `/api/ai/*`), ключ из UI (зашифрован).
5. **Structured output.** ИИ возвращает строгий JSON (риск/шаги/откат) — предсказуемый рендер в UI, не «простыня текста».

## Технологии

`Vue 3 + PrimeVue` → `FastAPI` → задачи в `RabbitMQ` → `Celery worker` (pg_dump/pg_restore/DDL) → `S3/MinIO`. `Redis` — Pub/Sub прогресса по WebSocket. Мета-БД `appdb` (SQLAlchemy async, Alembic). **ИИ:** OpenAI Chat Completions, ключ из настроек (Fernet-шифрование).

## Если что-то сломалось (troubleshooting)

| Симптом | Решение |
|---|---|
| Пустой экран / ошибка чанка после пересборки | Ctrl+Shift+R (hard reload) |
| ИИ-кнопка «выключен» | Настройки → ИИ → проверить ключ; статус должен быть «Включён» |
| Бэкап падает «клиент не установлен» | Настройки → PG Клиенты → установить версию сервера (v16) |
| Сервис упал | `docker compose up -d <service>` |
| Полный перезапуск | `docker compose up -d` (данные сохраняются в томах) |

Полный сброс демо-данных: `docker compose down -v && docker compose up -d --build`, затем заново создать организацию/сервер (см. историю команд) — **делать только при крайней необходимости, ключ и сервер придётся ввести заново**.
