# AGENTS.md — правила для Codex в проекте StageBridge AI

Этот файл — инструкции для OpenAI Codex. Читай его перед любой работой и **строго соблюдай guardrails**. Отвечай и комментируй код на русском языке.

## Что за проект

**StageBridge AI** — контрол-центр PostgreSQL с ИИ-слоем (хакатон OpenAI Build Week, категория Developer Tools).
- **Backend:** FastAPI (async SQLAlchemy + asyncpg), Celery + RabbitMQ, Redis pub/sub → WebSocket, Alembic, MinIO/S3, Fernet.
- **Frontend:** Vue 3 (`<script setup lang="ts">`) + PrimeVue + Pinia + Vite, vue-i18n (3 языка: kk/ru/en).
- **AI:** единый слой `backend/app/services/ai_service.py` → OpenAI Chat Completions (JSON mode), роуты в `backend/app/api/ai.py` под `/api/ai/*`. ИИ **только советует**, ничего не выполняет.
- **Запуск:** `docker compose up -d --build` → http://localhost. Демо-сервер `demopg` с примерными БД поднимается автоматически.

## Твоя задача

Она описана в **`docs/CODEX_TASKS.md`**. Делай ровно её, по шагам, с указанными критериями приёмки. Не расширяй скоуп самовольно.

## GUARDRAILS — чтобы ничего не сломать (обязательно)

### Границы: НЕ ТРОГАЙ эти файлы/области
- `backend/alembic/versions/**` — существующие миграции не редактировать. Новую миграцию создавать только если задача явно требует (эта — **не** требует изменения схемы БД).
- `docker-compose.yml`, `.env`, `.env.example`, `backend/app/config.py` (кроме добавления дефолта модели, если сказано в задаче).
- `backend/app/services/crypto.py`, `pg_connection.py`, `backup_service.py`, `structure_sync_tasks.py`, `db_admin_service.py` и всё, что связано с бэкапами/восстановлением/миграцией структуры — это критичная необратимая логика, её не касаться.
- Существующие рабочие AI-функции (`migration_plan`, `assistant`, `diagnostics_analysis`, `backup_risk`) — не менять их поведение.

### Принципы правок
1. **Только аддитивно.** Предпочитай новые функции/файлы. Правки существующих файлов — минимальные и точечные, строго в рамках задачи.
2. **Работай в отдельной ветке:** `git checkout -b feature/ai-query-advisor`. Не коммить в `main` напрямую. Коммить маленькими осмысленными коммитами на русском.
3. **i18n обязателен.** Любая новая строка UI добавляется во **все три** локали: `frontend/src/i18n/locales/ru.json`, `kk.json`, `en.json` — с сохранением строгого паритета ключей. Никаких хардкод-строк в шаблонах; только `t('namespace.key')`. Русский — исходный текст, kk — литературный казахский, en — грамотный английский.
4. **Следуй существующим паттернам.** Новая AI-функция — по образцу `ai_service.py` (async `(api_key, model, ...) -> dict`, `_chat(..., json_mode=True)`, `_safe_json`). Новый роут — по образцу `ai.py` (Pydantic In-модель, `_require_key`, вызов сервиса). UI — переиспользуй компонент `frontend/src/components/AiInsight.vue` (props: `label`, `endpoint`, `payload`, `sections`, `badgeField`).
5. **Модель GPT-5.6.** Требование хакатона — использовать GPT-5.6. В `_config()` (`api/ai.py`) и в `_chat` дефолт модели подними до `gpt-5.6` (значение всё равно берётся из настроек БД/`.env`, ключ задаёт пользователь в UI: Настройки → ИИ). Сохрани обратную совместимость: если API GPT-5.6 не принимает какой-то параметр (`response_format`/`temperature`/`max_tokens`) — аккуратно адаптируй `_chat`, не ломая существующие вызовы.
6. **Никаких секретов** в коде/логах/коммитах. Ключ OpenAI только через существующий механизм (БД, Fernet) — не хардкодить.

### Проверка перед завершением (обязательно прогнать)
- Frontend типы: `cd frontend && npx vue-tsc -b` — **0 ошибок**.
- Паритет локалей: количество ключей в ru/kk/en одинаковое; каждая новая `t('...')` присутствует во всех трёх.
- Сборка и подъём: `docker compose up -d --build backend frontend` — контейнеры поднимаются, `docker compose ps` без падений.
- Ручная проверка: открыть http://localhost → Мониторинг выбранного сервера → новая кнопка AI Query Advisor работает (при заданном ключе) и не ломает страницу при выключенном ИИ.
- Существующие 4 AI-функции и остальной UI по-прежнему работают.
